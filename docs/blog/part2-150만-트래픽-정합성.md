## 1. 12차 테스트 준비 — ASG 3대, 재고 130만

100만 정합성을 달성했으나 마지막으로 확인하고 싶은 것이 있었다.

11차에서 앱 CPU가 최대 90%에 달했다. 인스턴스 2대가 한계에 가까웠다는 뜻이다.

CPU가 포화되면 요청이 스레드 풀에서 대기하고 응답시간이 늘어난다. 인스턴스를 추가하면 ALB가 트래픽을 분산하므로 각 인스턴스의 CPU 부하가 줄고, 응답시간이 짧아져 TPS가 올라간다. 인스턴스를 늘리는 이유가 바로 이것이다.

인스턴스를 3대로 늘리면 어떤 변화가 생길까? 단순히 TPS가 오르는 것뿐 아니라,
더 많은 인스턴스가 동시에 Redis에 접근할 때 **시스템이 정합성을 여전히 보장하는지** 확인하고 싶었다.

12차 테스트 조건:

| 항목 | 값 |
|------|-----|
| 재고 | 130만 건 |
| 요청 | 150만 건 (초과 요청 20만 포함) |
| ASG | min=3, max=3 (3대 고정) |
| MAX_QUEUE_SIZE | 1,000,000 (변경 없음) |

## 2. 12차 결과 — 14만 건 유실 발견

테스트 종료 후 DB를 확인했다.

```sql
SELECT c.id, c.total_stock, c.status, COUNT(ph.id) AS success_count,
       c.total_stock - COUNT(ph.id) AS lost_count
FROM campaign c LEFT JOIN participation_history ph
  ON ph.campaign_id = c.id AND ph.status = 'SUCCESS'
WHERE c.id = 24 GROUP BY c.id, c.total_stock, c.status;
```

| id | total_stock | status | success_count | lost_count |
|----|-------------|--------|---------------|------------|
| 24 | 1,300,000 | CLOSED | 1,158,938 | **141,062** |

**141,062건이 유실됐다.**

API는 202(성공)를 반환했다. 사용자는 참여에 성공했다고 알고 있다. 그런데 DB에 없다.
PENDING 상태 기록도 없고, DLQ에도 없다. 복구가 불가능한 유실이었다.

## 3. Shared-Nothing 아키텍처의 테스트 어려움

이 버그는 단일 인스턴스 환경에서는 **절대 재현되지 않는다.**

이 시스템은 **Shared-Nothing 아키텍처**로 설계되어 있다.

Shared-Nothing이란 각 노드(인스턴스)가 메모리도 디스크도 공유하지 않고, 오직 외부 시스템(Redis, Kafka, DB)을 통해서만 소통하는 구조를 말한다. 인스턴스를 늘리면 처리량이 선형으로 증가하고, 특정 인스턴스가 죽어도 나머지가 서비스를 유지한다는 장점이 있다.

```
인스턴스 A ─┐
인스턴스 B ─┼──→ Redis (ElastiCache) ──→ Kafka ──→ Consumer ──→ RDS
인스턴스 C ─┘
```

그런데 이 구조가 테스트를 어렵게 만든다.

- 단위 테스트로는 Redis 연산 간 타이밍 문제를 재현할 수 없다
- 인스턴스 1대로 테스트해도 문제가 드러나지 않는다
- **Queue 상한에 닿는 순간에만 발생하는 버그라 실제 부하를 줘봐야 보인다**

11차까지는 인스턴스 2대였고, 처리량이 낮아 우연히 Queue가 상한(1M)에 닿지 않았다.
12차에서 인스턴스 3대로 늘리자 처리량이 증가해 Queue가 처음으로 상한에 닿았고, 그 순간 숨어있던 버그가 수면 위로 올라왔다.
2대였어도 Queue가 상한에 닿았다면 동일하게 발생했을 버그다. 3대가 특별한 게 아니라, 3대가 처리량을 높여서 상한 도달을 앞당긴 것뿐이다.

이것이 Shared-Nothing 분산 시스템의 어려움이다.
**로컬에서 아무리 열심히 테스트해도 분산 환경의 타이밍 문제는 실제 부하를 줘봐야만 보인다.**

## 4. 원인 분석 — 두 가지 문제를 동시에 발견

원인이 **두 가지**였다. 그리고 둘이 결합해서 복구 불가능한 유실을 만들었다.

### 원인 1: Queue 크기 부족

재고 130만인데 MAX_QUEUE_SIZE가 1M이었다.

API 요청 속도가 Bridge의 drain 속도보다 빠른 초반 구간에 Queue가 빠르게 쌓인다.
Queue가 1M 상한에 닿으면 이후 LPUSH가 실패한다.

```
API 적재 속도 > Bridge drain 속도
    → Queue 1M 상한 도달
    → 신규 LPUSH 실패
```

이 자체만으로도 유실이 발생하지만, 진짜 문제는 두 번째 원인과 결합했을 때다.

### 원인 2: DECR과 LPUSH가 원자적이지 않았다

수정 전 코드는 재고 차감과 큐 적재를 **두 개의 별도 Lua 스크립트**로 나눠서 실행했다.

```
[수정 전 — 두 번의 Redis 왕복]

1. check-decr-total.lua 실행 (1번째 왕복)
   → EXISTS active:campaign:{id} 체크
   → DECR stock:campaign:{id}      ← 재고 차감
   → GET  total:campaign:{id}
   → remaining 반환

2. push-queue.lua 실행 (2번째 왕복)
   → LLEN queue:campaign:{id} 체크
   → LPUSH queue:campaign:{id}     ← 큐 적재
```

1번과 2번 사이에 시간 간격이 존재한다.
Queue가 꽉 찬 상황에서 다음과 같은 일이 발생했다.

```
인스턴스 A: check-decr-total.lua 실행 → remaining = 500,000  (재고 차감 완료)
인스턴스 B: check-decr-total.lua 실행 → remaining = 499,999  (재고 차감 완료)
인스턴스 C: check-decr-total.lua 실행 → remaining = 499,998  (재고 차감 완료)
                ...
        (이 시점에 Queue가 1M 상한에 도달)
                ...
인스턴스 A: push-queue.lua 실행 → Queue FULL → LPUSH 실패
인스턴스 B: push-queue.lua 실행 → Queue FULL → LPUSH 실패
인스턴스 C: push-queue.lua 실행 → Queue FULL → LPUSH 실패
```

결과:
- 재고는 차감됐다 (DECR 성공)
- Queue에는 없다 (LPUSH 실패)
- API는 이미 202를 반환했다 (DECR 성공 시점에 응답)
- Kafka에도 없다, DB에도 없다
- **복구 경로가 없다** — PENDING 기록이 애초에 없으므로 ConsistencyJob도 탐지하지 못한다

이것이 141,062건 유실의 정확한 원인이었다.

## 5. 해결 — Lua 원자화와 ElastiCache CME 해시태그

두 가지를 동시에 적용했다.

### 수정 1: MAX_QUEUE_SIZE 1.5M으로 상향

목표 재고 150만에 맞춰 Queue 상한도 1.5M으로 올렸다.
Queue가 재고량보다 항상 여유 있게 유지되어야 overflow 자체를 방지할 수 있다.

```java
// RedisStockService.java
private static final long MAX_QUEUE_SIZE = 1_500_000;
```

**그런데 Queue 상한을 무작정 높이면 안 된다.** Redis는 순수 인메모리 저장소라, 큐에 쌓인 메시지 수만큼 RAM을 소비한다.

```
메시지 1건 ≈ userId + campaignId + sequence ≈ 100~150 bytes
1,500,000건 × 150 bytes ≈ 225 MB
```

이 프로젝트는 ElastiCache cache.t3.micro × 6 (3샤드) 구성이라 샤드당 가용 메모리가 약 555MB다. 225MB는 충분히 수용 가능한 수치였기 때문에 1.5M으로 결정했다. 만약 인스턴스 사양이 낮았다면 상한을 이렇게 높이기 어려웠을 것이다.

MAX_QUEUE_SIZE를 올릴 때 고려해야 할 트레이드오프는 세 가지다.

| 항목 | 내용 |
|------|------|
| **메모리 압박** | 더 많이 쌓일수록 ElastiCache RAM 소비 증가 |
| **드레인 지연** | Bridge가 100ms마다 RPOP하는 속도에 한계가 있다. 큐가 클수록 전체 처리 완료까지 더 오래 걸린다 |
| **장애 시 손실 범위** | Redis는 인메모리라 재시작/장애 시 큐에 쌓인 데이터가 소실될 수 있다. 큐에 많이 쌓여 있을수록 잃을 수 있는 메시지가 많아진다 |

마지막 항목이 중요하다. **큐가 많이 쌓여 있는 상태는 그 자체로 리스크다.** 큐는 처리 지연의 버퍼이지, 영구 저장소가 아니다. 이 시스템에서 큐가 최대 1.2M까지 쌓였다가 Bridge와 Consumer가 소화하며 줄어든 패턴은 정상 동작이지만, 만약 Consumer가 멈춘 상태에서 Redis가 재시작된다면 큐에 쌓인 메시지 전체가 날아간다.

그래서 Queue는 **일시적 피크 버퍼** 역할에 한정되어야 한다. 큐가 계속 늘어나고 줄지 않는다면 Consumer 처리량 부족이라는 신호다. Bridge의 동적 batchSize 조절과 Consumer concurrency 확장이 이 문제를 완화하는 구조적 대응이다.

### 수정 2: 단일 Lua 스크립트로 원자화

핵심 수정이다. 기존에 두 개로 분리됐던 Lua 스크립트를 `check-decr-enqueue.lua` 하나로 통합했다.

Redis는 Lua 스크립트를 **단일 명령으로 원자적으로 실행**한다.
스크립트 실행 중에는 다른 클라이언트의 명령이 끼어들 수 없다.

```lua
-- check-decr-enqueue.lua
-- KEYS[1] = active:campaign:{campaignId}                        -- 캠페인 활성 플래그
-- KEYS[2] = stock:campaign:{campaignId}                         -- 재고
-- KEYS[3] = total:campaign:{campaignId}                         -- 전체 재고 (sequence 계산용)
-- KEYS[4] = queue:campaign:{campaignId}                         -- Redis Queue
-- KEYS[5] = participated:campaign:{campaignId}:user:{userId}    -- 중복 참여 방지

-- 1. 캠페인 활성 여부 체크
if redis.call('EXISTS', KEYS[1]) == 0 then
    return {-999, 0}   -- 비활성 캠페인 → 400
end

-- 2. 중복 참여 체크
if redis.call('EXISTS', KEYS[5]) == 1 then
    return {-997, 0}   -- 이미 참여한 유저 → 409
end

-- 3. Queue 만원 체크 → 여기서 막히면 DECR 자체를 실행하지 않음 (핵심!)
if tonumber(redis.call('LLEN', KEYS[4])) >= tonumber(ARGV[1]) then
    return {-998, 0}   -- Queue Full → 429, 재고 차감 없음
end

-- 4. 재고 차감
local remaining = redis.call('DECR', KEYS[2])
if remaining < 0 then
    return {remaining, 0}   -- 재고 소진 → 400
end

-- 5. 마지막 재고 소진 시 캠페인 active flag 제거
if remaining == 0 then
    redis.call('DEL', KEYS[1])
end

-- 6. sequence 계산 후 큐 적재 + 참여 이력 동시 기록
local total = tonumber(redis.call('GET', KEYS[3])) or 0
local sequence = total - remaining
local message = '{"campaignId":' .. ARGV[2] .. ',"userId":' .. ARGV[3] .. ',"sequence":' .. sequence .. '}'
redis.call('LPUSH', KEYS[4], message)
redis.call('SET', KEYS[5], '1')   -- 영구 중복 참여 방지
return {remaining, total}
```

**핵심 변화**: Queue가 꽉 찼을 때의 흐름이 완전히 달라졌다.

```
[수정 전]
DECR 실행 → remaining 반환 → (별도 왕복) LLEN 체크 → Queue FULL → LPUSH 실패
→ 재고는 차감됐지만 큐에 없음 → 202 반환 → 유실

[수정 후]
LLEN 체크 → Queue FULL → -998 반환 → DECR 자체 미실행
→ 재고 차감 없음 → 429 반환 → 재시도 가능
```

DECR이 실행되지 않으므로 재고가 차감되지 않는다.
사용자는 429 응답을 받고, 재고는 온전히 보존된다.
**Partial Failure가 원천 차단된다.**

또 한 가지 세부 사항이 있다. Queue Full로 429를 반환할 때 `rateLimitService.release()`를 함께 호출한다.

```java
// ParticipationService.java
if (remaining == RedisStockService.QUEUE_FULL) {
    rateLimitService.release(campaignId, userId);   // rate limit 키 해제
    throw new QueueFullException(campaignId);
}
```

RateLimitService는 동일 유저의 10초 내 재요청을 차단하는 역할인데, Queue Full은 사용자 잘못이 아니라 일시적 시스템 상태다. 그래서 rate limit 키를 즉시 해제해 사용자가 10초를 기다리지 않고 바로 재시도할 수 있도록 설계했다.

단, 이 설계에는 비즈니스적 한계가 있다.

**Queue Full 시 선착순 보장이 부분적으로 깨진다.** 유저 A가 유저 B보다 먼저 요청했는데 둘 다 Queue Full로 429를 받았다고 하자. Bridge가 큐를 조금 비운 직후 유저 C가 새로 들어오면 C가 먼저 처리될 수 있다. A와 B는 재시도 타이밍에 따라 C보다 늦은 sequence를 받게 된다.

```
T=0   유저 A, B: 요청 → Queue Full → 429 (rate limit 해제)
T=1   Bridge: Queue 일부 drain
T=2   유저 C: 신규 요청 → 성공 → sequence 500,000
T=2   유저 A: 재시도 → 성공 → sequence 500,001   ← A가 먼저 신청했는데 늦은 순번
```

이를 완전히 해결하려면 다음과 같은 접근이 필요하다.

**방안 1 — Waiting Queue 도입**: Queue Full 시 요청을 즉시 거절하지 않고, 별도 대기열(예: Redis Sorted Set, score=요청 시각)에 넣는다. Bridge가 메인 Queue에 여유가 생기면 대기열에서 순서대로 꺼내 처리한다. 선착순이 보존되지만, 대기열 관리 로직과 대기 시간 안내 등 복잡도가 크게 올라간다.

**방안 2 — Queue를 충분히 크게 sizing**: Queue Full 자체가 발생하지 않도록 처음부터 재고량 이상으로 상한을 잡는다. 현재 설계 의도가 이것이다. Queue Full 경로는 예외 상황의 safety net이고, 정상 운영에서는 발동되지 않는 것이 목표다. 실제로 13차·14차에서 Queue가 최대 1.2M까지만 쌓였고 1.5M 상한에 닿지 않았다.

현재 시스템은 방안 2를 채택했다. Queue Full이 발생하지 않는 조건에서는 선착순이 완벽하게 보장된다. Queue Full이 발생하는 순간은 시스템이 감당할 수 있는 재고/Queue 한계를 초과한 극단적 상황이며, 그 경우에도 재고 정합성(총 DB 기록 = 총 재고)은 Lua 원자화로 보장된다.

그럼에도 방안 1이 비즈니스적으로 더 안전하다. 극단적 상황에서도 선착순 원칙을 지킬 수 있기 때문이다. 현재는 구현 복잡도를 이유로 방안 2를 선택했지만, 실제 프로덕션이라면 Waiting Queue를 도입해 선착순 보장을 완전히 확보하는 방향으로 개선할 의향이 있다.

### ElastiCache CME 문제 — 해시태그

Lua 스크립트에서 여러 키를 동시에 접근할 때 ElastiCache Cluster Mode Enabled(CME) 환경에서 문제가 생긴다.

Redis 클러스터는 키를 16,384개 슬롯에 해시로 분산 저장한다. 키 이름이 다르면 서로 다른 슬롯에 배치되는데, 단일 Lua 스크립트에서 다른 슬롯의 키를 동시에 접근하면 이 에러가 발생한다.

```
CROSSSLOT Keys in request don't hash to the same slot
```

해결책은 **해시태그**다. 키 이름에 `{campaignId}` 중괄호를 포함하면 Redis가 중괄호 안의 값만으로 슬롯을 계산한다. 같은 campaignId를 가진 키는 모두 같은 슬롯에 배치된다.

```
[수정 전 — 각자 다른 슬롯]
active:campaign:28    →  슬롯 A
stock:campaign:28     →  슬롯 B       ← Lua 멀티키 접근 시 CROSSSLOT 에러
queue:campaign:28     →  슬롯 C

[수정 후 — 해시태그로 동일 슬롯 보장]
active:campaign:{28}  ─┐
stock:campaign:{28}   ─┼→  모두 슬롯 K (28 기준으로 계산)
queue:campaign:{28}   ─┘   → Lua 스크립트 정상 실행
```

코드에서는 다음과 같이 키를 생성한다.

```java
// RedisStockService.java
private static final String ACTIVE_FLAG_KEY_PREFIX  = "active:campaign:{";
private static final String STOCK_KEY_PREFIX        = "stock:campaign:{";
private static final String TOTAL_KEY_PREFIX        = "total:campaign:{";
private static final String QUEUE_KEY_PREFIX        = "queue:campaign:{";
private static final String PARTICIPATED_KEY_PREFIX = "participated:campaign:{";

private String getQueueKey(Long campaignId) {
    return QUEUE_KEY_PREFIX + campaignId + "}";  // "queue:campaign:{28}"
}
```

그리고 또 하나의 부수 효과가 있었다. 기존에 두 번의 Redis 왕복(DECR용 + LPUSH용)이 필요했던 것이 Lua 한 번으로 줄었다. **Redis 왕복 2회 → 1회**, 레이턴시 개선까지 얻었다.

### T03 — 원자화 경로 별도 검증

13차와 14차에서 Redis Queue가 최대 1.2M까지만 쌓였다.
1.5M 상한에 닿지 않았으므로 **Lua 스크립트의 Queue full 경로(-998 반환)가 실제로 발동된 적이 없다.**

"Queue가 꽉 찼을 때 DECR이 실행되지 않고 429를 반환한다"는 것을 실제로 검증하려면 별도 테스트가 필요했다.

### 테스트 설계

- MAX_QUEUE_SIZE를 **500K로 임시 축소**
- 재고 100만, 요청 충분히 많게
- Queue가 500K에 닿으면 Lua가 -998을 반환하고 DECR을 실행하지 않아야 한다
- 검증 공식: **Redis 잔여재고 + DB success_count = total_stock**

### T03 결과

![T03 Redis Queue 500K 상한 도달](../../3-broker\ clustering\ 이후-\ 테스트\ 사진/장애\ 테스트/T03/스크린샷\ 2026-05-05\ 203715.png)

Queue가 500K 상한에 정확하게 닿으며 신규 적재가 멈췄다.

```bash
# Queue 상한 도달 후 Redis 잔여재고 확인
$ valkey-cli GET "stock:campaign:{43}"
"403524"

# Queue drain 완료 후 크기 확인
$ valkey-cli LLEN "queue:campaign:{43}"
(integer) 0
```

![T03 Redis 잔여재고 확인](../../3-broker\ clustering\ 이후-\ 테스트\ 사진/장애\ 테스트/T03/KakaoTalk_20260505_204147279.png)

![T03 DB 정합성 쿼리 결과](../../3-broker\ clustering\ 이후-\ 테스트\ 사진/장애\ 테스트/T03/KakaoTalk_20260505_204346809.png)

이 시스템은 Redis-first 구조라 API 경로에서 DB의 `current_stock`을 갱신하지 않는다. Queue가 꽉 찬 시점에 캠페인은 아직 CLOSED 처리되지 않았으므로 `current_stock`은 초기값(1,000,000)을 유지한다. 따라서 `total_stock - current_stock = 0`이 되어 아래 표의 `diff` 컬럼은 의미 없는 값이 된다. 실제 정합성 검증은 **Redis 잔여재고 + DB success_count = total_stock** 공식으로 한다.

```sql
SELECT c.id, c.total_stock, c.current_stock, COUNT(ph.id) AS success_count,
       (c.total_stock - c.current_stock) AS expected_success,
       COUNT(ph.id) - (c.total_stock - c.current_stock) AS diff
FROM campaign c LEFT JOIN participation_history ph
  ON ph.campaign_id = c.id AND ph.status = 'SUCCESS'
WHERE c.id = 43
GROUP BY c.id, c.total_stock, c.current_stock;
```

| id | total_stock | current_stock | success_count | expected_success | diff |
|----|-------------|---------------|---------------|------------------|------|
| 43 | 1,000,000 | 1,000,000 | 596,476 | 0 | 596,476 (※ current_stock 미갱신으로 의미 없음) |

- DB success_count = **596,476**
- Redis 잔여재고 = **403,524**
- 596,476 + 403,524 = **1,000,000 (total_stock)** 정확히 일치

숫자가 정확히 맞아떨어진다.

Queue가 꽉 찼을 때 Lua가 DECR을 실행하지 않았고, 재고 403,524는 차감되지 않은 채 Redis에 남아있다. DB에는 실제로 큐를 통과한 596,476건만 기록됐다. **어디서도 유실이 없다.**

원자화 경로가 의도한 대로 정확히 동작함을 확인했다.

### 발견한 허점

복구 후 재처리는 정상 동작했지만, 테스트 중 한 가지 허점을 발견했다.

마지막 재고가 소진될 때(`remaining==0`) 캠페인을 CLOSED 처리하기 위한 동기 DB 호출이 API 경로에 남아 있었다. Redis-first 구조로 전환했음에도 이 호출 하나가 RDS 장애 시 500으로 전파되고 있었다.

```
[수정 전]
remaining == 0
  → Redis active flag DEL  (Lua 내부에서 이미 처리)
  → DB campaign status = CLOSED  ← RDS 장애 시 500 전파

[수정 후]
remaining == 0
  → try-catch로 감싸고 ConsistencyJob fallback에 위임
  → Redis Lua가 이미 active flag를 삭제했으므로 신규 요청은 차단됨
```

### 자동 복구 확인

SG를 복구하자 Kafka lag이 빠르게 해소됐다. Consumer가 ack를 보류하고 있던 메시지들을 Kafka가 재전달했고, Consumer가 이를 정상 처리해 DB에 기록됐다. 별도의 수동 개입 없이 자동으로 복구됐다.

## 9. v1 대비 최종 성능 비교 — 246/s → 3,737/s

### v1이 멈춘 이유

v1은 10만 건 트래픽을 대상으로 **Kafka 파티션 수에 따른 처리량 vs 순서 보장 트레이드오프**를 실험한 프로젝트였다.

v1에서 발견한 핵심 사실이 두 가지 있었다.

**첫째**, 파티션을 1 → 3 → 5로 늘려도 TPS가 ~285에서 거의 정체됐다.

| 파티션 수 | DB Commit TPS |
|-----------|--------------|
| 1개 | 278/s |
| 3개 | 505/s |
| 5개 | 595/s |

Redis 도입 후에야 DB 병목(Row Lock)이 해소되며 파티션 확장 효과가 나타났다. 하지만 이때 TPS 상한이 595/s였고, 인프라는 단일 EC2 t3.large 1대였다.

**둘째**, 파티션을 늘리면 순서 보장이 불가능해진다.

v1에서는 이 문제를 해결하지 못해 **파티션 1개를 선택**했다. "처리량보다 공정성이 더 중요하다"는 결론이었다. 동시에 v1 문서에 다음 개선 방향을 적어뒀었다.

> *"Redis 기반 대기열 도입 — API 앞단에서 요청을 Redis LIST에 등록하고, 스케줄러가 시스템이 감당 가능한 수준만큼만 꺼내어 Kafka로 발행하는 구조"*

v3는 이 두 가지 문제를 모두 해결한 결과물이다.

---

### v1 → v3 구조 비교

| 항목 | v1 | v3 |

| 목표 트래픽 | 10만 건 | **150만 건** |

| 평균 TPS | 246~595/s | **~3,737/s** |

### 순서 보장 설계의 진화

v1에서 가장 까다로웠던 문제가 "파티션을 늘리면 순서가 깨진다"였다.

v1에서 파티션이 3개면 Consumer 스레드도 3개가 뜬다. 각 스레드가 DB commit을 완료하는 순서는 예측 불가능하다. 먼저 요청한 사람이 나중에 DB에 기록될 수 있다.

```
[v1 — 파티션 3개 시 순서 보장 불가]
Thread-0: 1번째 요청 처리 중 → DB commit (3번째 완료)
Thread-1: 2번째 요청 처리 중 → DB commit (1번째 완료)
Thread-2: 3번째 요청 처리 중 → DB commit (2번째 완료)
→ DB 기록 순서: 2번 → 3번 → 1번
```

v1은 이 문제를 파티션 1개로 회피했다. 근본 해결이 아니라 우회였다.

v3는 구조 자체를 바꿨다. **DECR 시점에 sequence를 계산해 메시지 안에 포함**한다.

```lua
-- check-decr-enqueue.lua
local remaining = redis.call('DECR', KEYS[2])
local total = tonumber(redis.call('GET', KEYS[3])) or 0
local sequence = total - remaining  -- DECR 순간에 확정
local message = '{"campaignId":..., "userId":..., "sequence":' .. sequence .. '}'
redis.call('LPUSH', KEYS[4], message)
```

이후 Kafka → Consumer → DB로 이어지는 파이프라인이 순서를 섞더라도 상관없다. 각 메시지가 이미 자신의 선착순 번호를 갖고 있기 때문이다. 파티션 10개, Consumer 스레드 10개로 병렬 처리해도 공정성은 깨지지 않는다.

v1에서 "파티션 늘리면 순서 보장 불가"라는 제약이 v3에서는 완전히 사라졌다. 덕분에 파티션 10개 + concurrency 10으로 처리량을 선형 확장할 수 있었다.

---

### v1 개선 방향이 v3에서 구현됐다

v1 문서에서 향후 개선 방향으로 언급했던 "Redis 기반 대기열"이 v3에서 그대로 구현됐다.

```
[v1 개선 방향으로 적어뒀던 것]
"API 앞단에 Redis LIST를 대기열로 구성, 스케줄러가 감당 가능한 만큼만 꺼내어 Kafka로 발행"

[v3 실제 구현]
API → Redis LPUSH (queue:campaign:{id})
ParticipationBridge @Scheduled 100ms → RPOP → Kafka publish
동적 batchSize: 큐 크기에 따라 500/1000/2000 자동 조절
```

구조적으로 동일하다. v1에서 설계 방향을 올바르게 잡았고, v3에서 실제로 구현해 150만 정합성을 달성했다.

### 병목은 해소될 때마다 다음 단계로 이동했다

각 차수에서 병목을 제거하면, 숨어있던 다음 병목이 드러나는 패턴이 반복됐다.

```
1~3차  DB Row Lock → HikariCP pending 950~980 → TPS 323/s 한계
  ↓
4차    API 경로에서 DB 완전 제거 (Redis-first)
  ↓
       HikariCP pending ≈ 0, 새 병목 드러남: Kafka Consumer 지연 (DB INSERT 1건씩)
  ↓
5~6차  Kafka 파티션/userId 키 최적화 → Consumer 지연 1.25s→200ms
  ↓
       Consumer 병목 해소, 새 병목 드러남: 브로커 처리량 (단일 브로커)
  ↓
7~8차  3-broker + 파티션 10개 → 브로커 병목 해소
  ↓
       브로커 병목 해소, 새 병목 드러남: 앱 CPU (t3.small 1대)
  ↓
9차    ASG 2대 수평 확장 → TPS +65% → 앱 CPU 분산
  ↓
10차~  배치 INSERT, Lua 원자화, Queue 최적화
  ↓
       최종 병목: 앱 CPU (t3.small 3대 한계, 97%)
```

이 패턴에서 배운 것은 **최적화는 항상 현재 병목에만 집중해야 한다**는 것이다.

4차 이전에 Kafka 파티션을 늘렸어도 TPS는 거의 오르지 않았을 것이다. DB가 병목이었으니까.
HikariCP가 pending 980이던 시절에 ASG를 추가해도 DB Lock 경합만 늘어났을 것이다.
**측정 → 병목 특정 → 그것만 제거 → 재측정** 사이클이 핵심이다.

최종 상태에서 병목이 앱 CPU라는 것은 좋은 신호다.
DB도 Kafka도 Redis도 병목이 아니라, **애플리케이션 연산 자체**가 한계라는 뜻이기 때문이다.
더 높은 TPS가 필요하다면 인스턴스를 추가하거나 더 큰 타입으로 교체하면 된다.
구조적 문제가 아니라 수평 확장으로 해결 가능한 상태다.

### 아키텍처 변화

```
[v1 — 초기 구조]
POST /api/campaigns/{id}/participation
  → DB SELECT FOR UPDATE   (재고 조회 + Lock 획득)
  → DB UPDATE stock - 1    (재고 차감)
  → DB INSERT              (참여 기록)
  → 200 반환

병목: DB Row Lock 경합 → HikariCP pending 950~980 → TPS 246/s


[v3 — 최종 구조]
POST /api/campaigns/{id}/participation
  → RateLimitService: SET NX EX 10  (10초 중복 차단)
  → check-decr-enqueue.lua 단일 원자 실행
     ├── active 체크
     ├── participated 체크 (영구 중복 방지)
     ├── Queue LLEN 체크  (Queue full → -998, DECR 미실행)
     ├── DECR             (재고 차감)
     ├── LPUSH            (큐 적재)
     └── SET participated (참여 이력)
  → 202 반환 (DB 미접촉)

ParticipationBridge (@Scheduled 100ms)
  → SMEMBERS active:campaigns
  → RPOP + Kafka publish (userId 파티션 키)
  → 동적 batchSize: <10K→500 / <100K→1K / >=100K→2K

Kafka Consumer (concurrency = 파티션 수 자동 감지)
  → jdbcTemplate.batchUpdate() INSERT IGNORE
  → 배치 실패 시 단건 폴백 → DLQ
  → transient failure 시 ack 보류 → Kafka 자동 재전달
```

v1은 요청 1건마다 DB를 3번 건드렸다. DB가 모든 것의 병목이었다.
v3는 API 경로에서 DB를 완전히 제거했다. Redis Lua 원자 실행 1번으로 모든 검증과 상태 변경이 끝난다.

| 차수 | 핵심 변경 | TPS | 병목 |

병목이 이동했다. v1에서는 DB Row Lock이 병목이었다. 지금은 **앱 CPU**가 병목이다.

### 장애 시나리오 전체 통과

| # | 시나리오 | 결과 |
|---|----------|------|
| T01 | 중복 요청 차단 (RateLimitService + participated 키) | ✅ |
| T02 | 재고 소진 400 반환 + 정합성 | ✅ |

### t3 Burstable 인스턴스와 CPU Credit 리스크

14차 테스트에서 앱 CPU가 97%에 달했다. 테스트는 문제없이 완료됐지만, 한 가지 잠재적 리스크가 있다.

t3 인스턴스는 **Burstable** 타입이다. 기준 CPU는 t3.small 기준 **20%** 이고, 나머지는 CPU Credit을 소진해 burst한다. Credit이 고갈되면 CPU는 자동으로 20%로 스로틀링된다.

14차는 약 6분짜리 테스트라 Credit 고갈이 발생하지 않았다.
하지만 장기간(수십 분 이상) CPU 97%를 유지하는 실제 프로덕션이라면 Credit이 고갈되어 TPS가 급락할 수 있다.

```
t3.small 기준 CPU: 20%
burst 가능 CPU:    100% (Credit 보유 시)
Credit 고갈 후:    20%로 강제 스로틀링
```

실제 프로덕션이라면 M5/C5 같은 Fixed Performance 인스턴스를 선택해야 한다.
이 프로젝트에서 t3를 사용한 이유는 비용 절감 목적이며, 성능 한계를 측정하고 개선하는 실험적 용도에 집중했기 때문이다.
인프라 선택 시 Burstable 타입은 평상시 CPU가 낮고 간헐적으로 burst가 필요한 워크로드에 적합하다.
지속적인 고부하 처리에는 Credit 소진 리스크를 감수해야 한다는 점을 인지해야 한다.

### 반복 실험에서 배운 것

이 프로젝트에서 가장 중요하게 배운 것은 **측정 → 원인 → 수정 → 재실험** 사이클이다.

12차 유실 141K는 Grafana 13개 패널 전체가 green인 상태에서 발생했다.
API TPS 정상, 5xx 없음, CPU 정상, HikariCP pending 거의 0. 모든 지표가 이상 없었다.
**DB를 직접 COUNT 쿼리로 찍어봤기 때문에 발견했다.** 인프라 지표는 "지금 시스템이 살아있는가"를 알려줄 뿐, "비즈니스 데이터가 올바르게 처리됐는가"는 데이터를 직접 확인해야 알 수 있다.

```sql
-- 이 쿼리 하나가 13개 Grafana 패널보다 더 중요한 정보를 담고 있었다
SELECT c.total_stock - COUNT(ph.id) AS lost_count
FROM campaign c LEFT JOIN participation_history ph
  ON ph.campaign_id = c.id AND ph.status = 'SUCCESS'
WHERE c.id = 24;
-- → 141,062
```

그리고 Shared-Nothing 분산 시스템은 단위 테스트로 검증할 수 없는 영역이 반드시 존재한다. 실제 분산 환경에서 실제 부하를 줘봐야만 보이는 문제가 있다.

### 부하 테스트 설계에서 배운 것

**shared-iterations를 선택한 이유**는 정합성 검증의 기준이 되려면 총 요청수가 정확해야 하기 때문이다.
`constant-vus`는 duration 동안 계속 요청하므로 총 요청수가 매번 달라진다. 총 요청수가 불확정이면 "몇 건이 유실됐는지"의 분모 자체가 흔들린다.
`shared-iterations`로 160만 건을 고정하면 테스트 종료 후 `DB success_count + 실패(400/429)건수 = 160만` 등식으로 정합성을 검증할 수 있고, 13차·14차처럼 재현도 가능해진다.

**VUS가 높다고 TPS가 높아지지 않는다.** Little's Law — `TPS = VUS / 응답시간`.
VUS=3,000이고 응답시간 746ms이면 TPS ≈ 4,020/s, 실측 3,737/s와 거의 일치한다.
CPU가 포화 상태라면 VUS를 아무리 올려도 응답시간만 길어지고 TPS는 그대로다.

| VUS | 응답시간 | TPS | 앱 CPU | 적합한 용도 |
|-----|---------|-----|--------|------------|
| 500 | ~15ms | ~1,300/s | ~30% | 장애 테스트 — 부하 유지하면서 Kafka lag, 5xx 관찰 |
| 1,500 | ~300ms | ~2,500/s | ~50% | 안정 운영 시뮬레이션 |
| 3,000 | ~746ms | ~3,737/s | ~97% | 최대 한계 측정 |

장애 테스트에 VUS=500을 쓴 이유가 있다. CPU 97%에서 Kafka 브로커를 내리면 장애 반응인지 포화 반응인지 구분이 안 된다. VUS=500에서 CPU 30% 수준이면 여유 있는 상태에서 장애를 주입하고 TPS 변화와 5xx를 깔끔하게 관찰할 수 있다.
